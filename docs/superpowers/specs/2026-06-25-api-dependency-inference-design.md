# API 接口依赖自动推断设计

## 背景

当前通过接口生成测试用例时，用户手动选择接口，AI 根据选中的接口生成用例。但常见的 CRUD 场景中，编辑、删除、详情等接口都依赖"新增"接口来获取记录 ID。用户需要手动把新增接口也选上，否则生成的用例无法串联。

## 目标

1. 用户选中编辑/删除/详情等接口时，系统自动推断并补全依赖的"新增"接口
2. AI 生成时能理解接口间的依赖关系，生成"先新增 → 再操作"的完整流程
3. 新增接口返回的记录 ID 通过 `post_actions` + `{{变量}}` 机制自动传递给后续接口

## 设计

### 一、依赖推断算法

在后端 `backend/app/routes/testcase.py` 的生成接口中，构建 API 上下文之前执行依赖推断。

#### 1.1 识别"被依赖"接口（候选新增接口）

遍历同一 `api_project_id` 下的所有接口，匹配以下条件：

```
method == "POST" 且 path 中不包含 /{ 或 /:
```

即：POST 方法 + 非路径参数化的接口，视为候选"新增"接口。

#### 1.2 识别"依赖方"接口

遍历用户选中的接口，满足以下任一条件即为依赖方：

- **路径参数匹配**：路径中包含 `/{xxx}` 或 `/:xxx` 模式，且参数名包含 `id`（不区分大小写），如 `/{id}`, `/{userId}`, `/:recordId`
- **请求参数匹配**：请求 body 或 parameters 中存在 `id` 字段（字段名包含 `id` 后缀，如 `user_id`, `recordId`），且该字段不出现在响应 schema 的顶层字段中

路径参数匹配优先级高于请求参数匹配（路径中有 id 参数的接口几乎一定依赖前置接口）。

#### 1.3 自动补全

如果存在依赖方但用户未选中任何候选新增接口，则自动把候选新增接口加入生成上下文。

**多候选接口的选择规则**：如果存在多个候选新增接口（如 `POST /api/users` 和 `POST /api/orders`），按以下优先级匹配：

1. **路径前缀匹配**：候选接口的路径前缀与依赖方的路径前缀相同。例如依赖方是 `PUT /api/users/{id}`，候选 `POST /api/users` 的前缀 `/api/users` 匹配，选中
2. **如果无前缀匹配**：选择路径层级最接近的候选（路径 segments 数量差最小的）
3. **如果仍无法区分**：全部加入（让 AI 判断哪个合适）

### 二、增强系统提示词

在 `model_utils.py` 的系统提示词中，当检测到有依赖补全的接口时，追加以下指导：

```
## 接口依赖关系

以下接口存在依赖关系，被标记为 [依赖补全] 的接口是系统自动添加的前置依赖：

[接口列表，标注依赖关系]

请生成测试用例时遵循以下规则：
1. 对于依赖新增接口的场景，在 preset_conditions 中生成一个 api_call 步骤调用新增接口
2. 在新增接口步骤上添加 post_actions 提取返回的记录 ID（如 $.data.id）
3. 在后续编辑/删除/详情步骤的 body 或路径参数中，使用 {{record_id}} 引用该 ID
4. 每个依赖场景至少生成一个成功用例
```

### 三、后端生成流程修改

文件：`backend/app/routes/testcase.py`

在 `POST /{session_id}/testcases` 接口中：

1. 在构建 API 上下文之前，调用新函数 `infer_endpoint_dependencies(selected_endpoint_ids, api_project_id)`
2. 该函数返回需要补全的接口 ID 列表
3. 把补全的接口 ID 追加到 `selected_endpoint_ids`
4. 在构建 API 上下文时，为补全的接口添加 `[依赖补全]` 标记
5. 该标记传递给 AI，让 AI 知道哪些是自动添加的前置依赖

### 四、涉及修改的文件

| 文件 | 修改内容 |
|------|----------|
| `backend/app/routes/testcase.py` | 新增 `infer_endpoint_dependencies()` 函数；在生成接口中调用 |
| `backend/utils/model_utils.py` | 系统提示词中追加依赖关系指导 |
| `backend/db/models.py` | 无需修改（现有字段足够） |

### 五、不涉及的内容

- 不修改前端（静默加入，用户无感知）
- 不修改测试用例数据模型（依赖关系通过步骤中的变量传递表达）
- 不修改执行引擎（现有的 `post_actions` + `{{变量}}` 机制已支持）
