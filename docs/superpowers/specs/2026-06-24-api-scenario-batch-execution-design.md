# 接口场景批量执行与最新结果展示设计

日期：2026-06-24

## 背景

接口场景测试工具已有接口项目、接口列表、场景列表、单场景执行和场景执行结果历史能力。当前需要增强场景列表的回归执行体验：支持批量执行场景，并在每个场景卡片下直接展示最新执行结果。

## 目标

- 场景列表支持多选、全选、批量执行选中场景。
- 支持一键执行当前项目下全部场景。
- 批量执行按场景列表顺序串行执行。
- 某个场景失败或异常时不中断后续场景，最终汇总通过和失败数量。
- 每个场景卡片展示最新执行状态和最近执行时间。
- 批量执行完成后刷新场景卡片状态、执行历史缓存和当前选中场景的结果详情。

## 非目标

- 不实现后台任务队列、进度轮询或页面刷新后恢复批量执行进度。
- 不实现并发批量执行。
- 不新增复杂报告导出能力。
- 不改变现有单场景执行语义。

## 推荐方案

采用后端批量执行接口方案。

新增项目级批量执行接口，由后端统一校验权限、确定执行顺序、串行执行场景、写入执行结果并返回汇总。前端负责选择场景、触发批量执行、展示汇总和刷新状态。

该方案比纯前端循环更可靠，批量执行语义更集中；同时避免引入后台任务系统，控制实现复杂度。

## 后端设计

### 新增接口

新增接口建议：

`POST /api-test/projects/{project_id}/scenarios/run-batch`

请求体：

```json
{
  "scenario_ids": [1, 2, 3],
  "run_all": false
}
```

字段语义：

- `scenario_ids`: 指定要执行的场景 ID 列表，用于多选批量执行。
- `run_all`: 是否执行当前项目下全部场景。

规则：

- `run_all=true` 时执行项目下全部场景，忽略或不要求 `scenario_ids`。
- `run_all=false` 时必须提供非空 `scenario_ids`。
- 后端按场景列表顺序执行，即与现有场景列表排序保持一致。
- 如果 `scenario_ids` 中包含不属于当前用户或当前项目的场景，返回错误，避免用户误以为已执行无效场景。
- 项目下没有可执行场景时返回 `total=0`，由前端提示“当前项目暂无可执行场景”。

响应体数据：

```json
{
  "total": 3,
  "passed": 2,
  "failed": 1,
  "results": [
    {
      "scenario_id": 1,
      "scenario_name": "登录成功流程",
      "record_id": 10,
      "passed": true,
      "created_at": "2026-06-24T10:00:00+08:00",
      "result": {}
    }
  ]
}
```

### 执行流程

1. 校验项目存在且属于当前用户。
2. 查询要执行的场景集合：
   - 全部执行：查询项目下全部场景。
   - 指定执行：查询项目下指定 ID 场景，并校验数量与请求 ID 一致。
3. 按现有列表顺序串行执行。
4. 每个场景复用现有 `run_scenario()` 逻辑。
5. 每个场景执行后写入一条 `ApiScenarioResult`。
6. 每个场景只保留最近 10 条执行历史。
7. 单个场景断言失败或执行异常均记为失败，不阻断后续场景。
8. 所有场景执行完成后返回汇总。

### 异常记录

如果某个场景执行过程抛异常，后端仍为该场景写入失败记录。失败记录的 `result` 至少包含：

```json
{
  "passed": false,
  "error": "错误信息",
  "steps": []
}
```

这样前端可统一按 `ApiScenarioResult` 展示最新状态。

## 前端设计

### 场景选择

在场景列表区域增加选择能力：

- 每个场景卡片左侧显示 Checkbox。
- 列表顶部增加操作区：
  - 全选
  - 取消选择
  - 批量执行
  - 执行全部
- `批量执行` 仅在已选择至少一个场景时可用。
- `执行全部` 不依赖勾选状态，执行当前项目下全部场景。

### 最新结果展示

每个场景卡片显示最新执行状态：

- 无结果：灰色 `未执行`
- 最新结果通过：绿色 `通过` + 最近执行时间
- 最新结果失败：红色 `不通过` + 最近执行时间

最新结果数据来源：

1. 加载项目场景后，为场景加载最新执行结果。
2. 单场景执行完成后，立即更新该场景状态。
3. 批量执行完成后，根据批量接口返回结果更新对应场景状态。

初版可沿用现有 `GET /scenarios/{id}/results?limit=1` 逐个加载最新结果。若后续场景数量很多，再新增项目级最新结果查询接口优化性能。

### 批量执行交互

批量执行期间：

- `批量执行` 和 `执行全部` 按钮显示 loading。
- 禁止重复触发批量执行。
- 禁用单场景执行按钮，避免同一场景或同一环境被并发执行。

执行完成后显示汇总消息，例如：

`批量执行完成：共 8 个，通过 6 个，不通过 2 个`

### 结果详情联动

当前选中的场景保持不变。

如果批量执行结果包含当前选中场景：

- 将右侧结果面板切换到该场景的最新执行结果。
- 同步更新该场景的执行历史缓存。

如果当前选中场景不在本次批量范围内：

- 不改变右侧结果面板。

## 数据结构影响

现有 `ApiScenarioResult` 模型已经能保存场景 ID、项目 ID、场景名称快照、是否通过、结果详情和用户 ID。本需求不需要新增数据库表或字段。

前端类型需要增加批量执行请求和响应类型，或在 API service 层使用局部类型。

## 权限与安全

- 批量执行接口必须校验 `ApiProject.user_id`。
- 执行的每个 `ApiScenario` 必须属于当前项目和当前用户。
- 非法场景 ID 返回错误，不静默忽略。
- 不允许通过批量接口执行其他用户或其他项目的场景。

## 验证计划

- 前端运行 `npm run lint`。
- 后端运行 Python 语法或模块导入检查。
- 手动验证：
  1. 场景卡片无历史时显示“未执行”。
  2. 单场景执行后卡片显示“通过/不通过 + 时间”。
  3. 勾选多个场景后批量执行，汇总数量正确。
  4. 执行全部按列表顺序串行执行。
  5. 批量中某场景失败或异常后仍继续执行后续场景。
  6. 当前选中场景参与批量执行时，结果详情自动刷新。

## 后续扩展

如果未来批量执行耗时较长或场景数量很大，可在当前接口语义基础上升级为后台任务模式，新增任务状态和进度轮询接口。当前设计不引入该复杂度。
